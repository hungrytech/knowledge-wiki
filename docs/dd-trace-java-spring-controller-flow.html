01 · request lifecycle

한 번의 HTTP 요청에서 일어나는 일

Controller span은 요청의 root가 아니다. 일반적으로 Servlet/container instrumentation이 만든 HTTP server context가 먼저 존재하고, HandlerAdapterInstrumentation이 그 활성 문맥 아래에 controller span을 추가한다. Spring이 mapping을 확정한 뒤에는 route와 handler method를 span resource에 반영한다.

02 · class-load-time instrumentation

Controller가 호출되기 전: agent는 어디를 바꾸나

여기서 핵심은 Spring의 bean proxy도, source-code weaving도 아니다. JVM이 application class를 정의하기 직전, Java Instrumentation의 ClassFileTransformer가 원본 byte[]를 받아 바꾼 class bytes를 JVM에 돌려주는 흐름이다. Datadog은 이 transformer를 Byte Buddy로 조립하고, Spring MVC module은 “어떤 type/method에 어떤 Advice를 붙일지”만 선언한다.

ASM을 어디까지 봐야 하나? 이 Spring module은 org.objectweb.asm.ClassReader를 직접 호출하지 않는다. module이 선언한 것은 Byte Buddy의 net.bytebuddy.asm.Advice다. 하지만 Advice가 최종적으로 수행하는 일은 class-file method body의 instruction stream에 진입·종료 코드를 합성하는 것이며, Byte Buddy의 ASM visitor 계층이 그 저수준 변환을 담당한다. 즉 Datadog의 integration 작성 surface는 Byte Buddy, 실제 bytecode 형식의 바닥은 ASM/JVM class-file이라고 보면 된다.

C. Spring MVC module: type matcher → method matcher → Advice

HandlerAdapterInstrumentation.java · 실제 선언
@AutoService(InstrumenterModule.class)
class HandlerAdapterInstrumentation
  extends InstrumenterModule.Tracing
  implements Instrumenter.ForTypeHierarchy,
             Instrumenter.HasMethodAdvice {

  hierarchyMarkerType() {
    return "org.springframework.web.servlet
      .HandlerAdapter";
  }

  hierarchyMatcher() {
    return implementsInterface(
      named(hierarchyMarkerType()));
  }

  methodAdvice(transformer) {
    transformer.applyAdvice(
      isMethod().and(isPublic())
        .and(nameStartsWith("handle"))
        .and(takesArgument(0,
          named("javax.servlet.http.HttpServletRequest")))
        .and(takesArguments(3)),
      "…$ControllerAdvice");
  }
}

03 · handler adapter controller advice

Controller span이 만들어지는 정확한 지점

실제 controller method 호출은 HandlerAdapter가 수행한다. 따라서 Agent는 HandlerAdapter.handle*의 enter/exit에 Advice를 연결한다. enter에서는 route를 parent server span에 반영하고, 활성 span이 있으면 새 controller span을 시작해 scope에 attach한다.

부모 HTTP span을 route로 보강

request attribute DD_CONTEXT_ATTRIBUTE에서 Context를 얻고, DECORATE.onRequest(parentSpan, request, request, rootContext())를 호출한다. decorator는 HandlerMapping.BEST_MATCHING_PATTERN_ATTRIBUTE가 있으면 HTTP resource를 예: GET /orders/{id}로 설정한다.

중복 방지 키를 계산

HandlerMethod라면 handler bean의 runtime class name, 그렇지 않으면 handler class name으로 dd.handler.span.<class> request attribute 키를 만든다. 이미 Context가 있으면 새 span을 만들지 않고 기존 Context를 attach한다.

controller span을 생성·attach

startSpan("spring-web-controller", DECORATE.spanName())가 spring.handler operation name의 measured span을 만든다. DECORATE.onHandle이 HandlerMethod의 declaring class + method name을 resource로 설정하고, span 자체는 request attribute에 저장된다.

종료 시 scope를 닫고 span finish

Advice exit에서 throwable을 decorator에 전달한다. async continuation 플래그가 없다면 beforeFinish 뒤 span.finish()한다. 즉 scope 종료와 span 종료는 구분되며, 비동기일 때 finish 시점은 뒤로 미뤄진다.

04 · timeline cases

동기, CompletionStage, 예외의 차이

일반적인 synchronous request

CompletionStage 반환: controller span은 즉시 끝나지 않는다

InvocableHandlerMethodInstrumentation은 invokeForRequest의 반환값이 CompletionStage인지 검사한다. request attribute에서 controller span을 찾아 .continue = true로 표시하고, 반환 stage를 whenComplete(AsyncResultExtensions.finishSpan(span))로 감싼다. 따라서 HandlerAdapter exit은 scope만 닫고, 실제 finish는 future completion에서 일어난다.

이 코드는 Servlet 기반 Spring MVC의 CompletionStage 반환 경로다. WebFlux/reactive controller, Kotlin coroutine, deferred result 등은 별도 instrumentation/version 조건을 확인해야 하며 같은 타임라인이라고 가정하면 안 된다.

예외는 두 층에서 관찰된다

ControllerAdvice.stopSpan은 @Advice.Thrown throwable이 있으면 DECORATE.onError(span, throwable)을 호출하고 span을 finish한다. 또 DispatcherServletInstrumentation은 processHandlerException*에 Advice를 적용한다. 여기서는 active span에 stack trace를 기록하지만, HTTP server decorator의 status code 정책(예: 5xx)에 따라 error 상태를 유지/결정한다.

따라서 “예외가 던져졌다”와 “trace UI에서 span이 error로 판정된다”는 별도 문제다. @ControllerAdvice가 예외를 4xx로 변환하는 애플리케이션은 예상 상태 코드를 포함해 검증해야 한다.
05 · semantic trace tree

Datadog에서 기대할 수 있는 span 관계

01

route ≠ handler resource

HTTP server span은 BEST_MATCHING_PATTERN_ATTRIBUTE로 route를 갖고, controller span은 onHandle에서 Java class/method 기반 resource를 갖는다.

02

span key는 request에 저장

같은 handler에 대해 이미 만든 span/Context가 있으면 재사용하여 중복 생성을 막는다.

03

DB·client span은 별도 module

JDBC/HTTP client instrumentation이 활성화되어야 controller의 child span으로 연결된다. Controller Advice가 그것들을 직접 생성하지는 않는다.

06 · operational boundaries

이 도식이 보장하지 않는 것

  • 정확한 support matrix: Spring, Servlet API (javax vs jakarta), JDK, dd-trace-java release마다 matcher와 integration 지원 범위를 확인해야 한다.
  • 표준 trace 전파: inbound trace headers에서 server context를 추출하는 계층은 이 Spring MVC module 바깥의 HTTP/Servlet instrumentation이다. 이 문서는 controller child span 생성 지점에 초점을 맞춘다.
  • 다른 agent 병용: Datadog은 다른 APM/tracing Java agent와 동시 로딩을 지원하지 않는다. Byte Buddy/ASM transformer 충돌은 실제 target JDK·framework 조합에서 시험해야 한다.
  • 수집 데이터: URL, header, query, body tagging 설정은 PII/secret을 노출할 수 있다. span 구조 이해와 수집 정책 활성화는 별도 의사결정이다.
