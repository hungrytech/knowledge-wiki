Chapter 02

Spring 기동: annotation을 endpoint로 바꾸는 시점

  1. @EnableKafka는 KafkaListenerConfigurationSelector를 통해 bootstrap configuration을 늦게 import한다.
  2. KafkaBootstrapConfiguration이 infrastructure bean으로 KafkaListenerAnnotationBeanPostProcessor와 KafkaListenerEndpointRegistry를 등록한다.
  3. 각 application bean 초기화 뒤 KLABPP가 target class와 method에서 @KafkaListener를 찾는다.
  4. MethodKafkaListenerEndpoint에 method와 annotation metadata를 담고 registrar에 등록한다.
  5. 모든 singleton 준비 후 registrar가 endpoint별 factory를 해결하여 registry에 concrete container 생성을 요청한다.
KafkaListenerAnnotationBeanPostProcessor.java · actual source capturelines 385–427
@Override
public Object postProcessAfterInitialization(final Object bean, final String beanName) throws BeansException {
  buildEnhancer();
  if (!this.nonAnnotatedClasses.contains(bean.getClass())) {
    Class<?> targetClass = AopUtils.getTargetClass(bean);
    Collection<KafkaListener> classLevelListeners = findListenerAnnotations(targetClass);
    Map<Method, Set<KafkaListener>> annotatedMethods = MethodIntrospector.selectMethods(targetClass,
      (MethodIntrospector.MetadataLookup<Set<KafkaListener>>) method -> {
        Set<KafkaListener> listenerMethods = findListenerAnnotations(method);
        return (!listenerMethods.isEmpty() ? listenerMethods : null);
      });
    boolean hasClassLevelListeners = !classLevelListeners.isEmpty();
    boolean hasMethodLevelListeners = !annotatedMethods.isEmpty();
    if (!hasMethodLevelListeners && !hasClassLevelListeners) {
      this.nonAnnotatedClasses.add(bean.getClass());
      this.logger.trace(() -> "No @KafkaListener annotations found on bean type: " + bean.getClass());
    }
    else {
      if (hasMethodLevelListeners) {
        for (Map.Entry<Method, Set<KafkaListener>> entry : annotatedMethods.entrySet()) {
          Method method = entry.getKey();
          for (KafkaListener listener : entry.getValue()) {
            processKafkaListener(listener, method, bean, beanName);
          }
        }
        this.logger.debug(() -> annotatedMethods.size() + " @KafkaListener methods processed on bean '"
          + beanName + "': " + annotatedMethods);
      }
      if (hasClassLevelListeners) {
        Set<Method> methodsWithHandler = MethodIntrospector.selectMethods(targetClass,
          (ReflectionUtils.MethodFilter) method ->
            AnnotationUtils.findAnnotation(method, KafkaHandler.class) != null);

        if (methodsWithHandler.isEmpty()) {
          throw new IllegalStateException("No Kafka listener methods in bean: " + bean);
        }

        List<Method> multiMethods = new ArrayList<>(methodsWithHandler);
        processMultiMethodListeners(classLevelListeners, multiMethods, targetClass, bean, beanName);
      }
    }
  }
  return bean;
}

    String beanName) {
  try {

    endpoint.setMethod(methodToUse);

@Override
public void start() {
  for (MessageListenerContainer listenerContainer : getListenerContainers()) {
    startIfNecessary(listenerContainer);
  }
  this.running = true;
}

private void startIfNecessary(MessageListenerContainer listenerContainer) {
  if ((this.contextRefreshed && this.alwaysStartAfterRefresh)
      || listenerContainer.isAutoStartup()) {
    listenerContainer.start();
  }
}
Late registration 함정. context refresh 뒤 endpoint를 등록하면 SmartLifecycle 계약 때문에 기본적으로 즉시 시작될 수 있다. 단지 autoStartup=false만 보고 안전하다고 판단하지 말고 registry의 alwaysStartAfterRefresh를 확인한다.

default phase는 Integer.MAX_VALUE - 100이고 auto-startup 기본값은 true다. DB migration, cache warmup, downstream readiness가 listener보다 먼저 끝나야 한다면 해당 SmartLifecycle component의 phase를 더 이르게 설계한다.

Chapter 04

Concurrency는 thread 수가 아니라 consumer topology다

KMLC

KafkaMessageListenerContainer 하나는 하나의 KafkaConsumer와 하나의 consumer thread를 가진다. 한 partition 내부 ordering은 이 직렬 처리 경계 위에 있다.

CMLC

ConcurrentMessageListenerContainer는 child KMLC를 N개 만들고 start/stop/pause/metrics를 집계하는 parent다. concurrency=N은 일반적으로 consumer N개다.

입력 모델partition을 누가 배정하는가concurrency보다 적은 partition일 때
topics / topic patternKafka consumer group coordinator + assignoridle consumer가 가능하다.
explicit TopicPartitionOffsetCMLC가 child KMLC에 나눈다container가 concurrency를 partition 수에 맞춰 낮춘다.

따라서 처리 병렬성은 단순한 설정값이 아니라 min(concurrency, assigned partitions)에 가까운 결과다. 여러 topic과 기본 RangeAssignor 조합에서는 원하는 분배가 나오지 않을 수 있다.

thread-safety. concurrent listener는 같은 bean instance를 여러 consumer thread에서 호출한다. listener와 공유 deserializer instance 모두 thread-safe인지 확인한다. thread-local을 쓰면 ConsumerStoppedEvent를 cleanup hook으로 설계한다.

개념적으로 loop는 pending commit/seek/rebalance/pause 상태를 먼저 처리하고, doPoll()에서 records를 받아 record 또는 batch adapter로 dispatch한다. record listener는 각 record에 대해 adapter의 onMessage(record[, acknowledgment][, consumer])를 호출하고, adapter가 `InvocableHandlerMethod`를 통해 user method를 실행한다.

consumer thread 경계. nack(), partial batch ack, Consumer API의 position/commit/seek와 직접 결합되는 동작은 listener를 호출한 consumer thread 경계를 넘기면 안 된다. 비동기 executor로 넘긴 작업이 consumer 또는 transaction context를 자동 상속한다고 가정하지 않는다.
Chapter 06

Offset · Error · Transaction: 재처리 의미를 결정하는 세 겹

AckMode를 먼저 선택한다

modecommit boundary주의점
RECORD각 record listener 정상 반환 뒤commit 빈도가 늘 수 있다.
BATCHpoll batch 전체 처리 뒤기본값; batch 일부 실패는 재처리 범위를 바꾼다.
TIME/COUNTbatch 처리 + 시간/수량 조건더 큰 duplicate window를 이해한다.
MANUALack 후 batch semanticslistener 책임이 커진다.
MANUAL_IMMEDIATEconsumer thread에서 ack 즉시호출 thread가 중요하다.

컨테이너 offset 모델을 쓸 때 auto commit은 대개 false여야 한다. Spring Kafka 2.3+는 명시하지 않으면 이를 false로 설정한다. asyncAcks는 순서 밖 manual ack를 허용하지만 gap에서 consumer pause/commit 지연을 일으킬 수 있으며 nack()와 함께 쓸 수 없다.

Error handler와 transaction은 같은 retry가 아니다

Non-transactional

DefaultErrorHandler + bounded BackOff + recoverer/DLT의 조합을 명시한다. 무한 retry는 poison record가 partition을 계속 막는 선택이다.

Transactional

listener 성공 시 offsets는 sendOffsetsToTransaction으로 transaction에 결합될 수 있다. 예외를 error handler가 삼키면 rollback을 기대한 경로가 사라질 수 있다.

EOS의 정확한 범위. Spring Kafka 3.0+ EOS V2에서 read-process-write의 write는 exactly once로 보호될 수 있지만, 외부 DB 호출이나 user business code가 “딱 한 번” 수행된다는 일반 보장은 아니다. idempotency/outbox/transaction boundary를 별도로 설계한다.
